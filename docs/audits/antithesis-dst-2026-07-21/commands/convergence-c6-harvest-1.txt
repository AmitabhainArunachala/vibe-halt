## Provenance (playbook source priority 1: real AI-framework defect)

langchain-ai/langgraph issue #7417 — "Long tool calls (~180s+) silently
re-executed from checkpoint on LangGraph Cloud" (2026-04-05; versions
1.1.3-1.1.6, Cloud Plus). Mechanism verified from the issue body:
BG_JOB_HEARTBEAT hardcoded 120s; queue sweep every 240s marks
long-running tasks stale; CancelledError in ALL_RETRIABLE_EXCEPTIONS
restarts swept runs from pending WHILE the original still executes;
both complete; duplicate tool invocations with identical arguments.

2. Workload: corpus-stale-redispatch (crates/vh-cli/src/workloads/
   corpus.rs, VB-007 section). Delay-only palette — nothing is ever

   detail names each over-applied task and count.

   divergent 0; 9 fault-light universes PASS (vacuous-failure doctrine

     always-failures: 91 universe(s); divergent: 0; ...  (exit 1)

   exit 1 + one FINDINGS verdict + anchored oracle:exactly_once_dispatch.

## Increment 3 (same PR): VB-009 transient-fatal abort

Provenance: OpenHands/OpenHands issue #12064 (2025-12-16; FIXED by PR
#12117) — 502 Bad Gateway surfaces as litellm.APIError, missing from
LLM_RETRY_EXCEPTIONS in openhands/llm/llm.py; retry logic doesn't
recognize it; agent controller catches the unhandled exception and the
agent crashes, abandoning the session. Verified from the issue at point
of use.

Reduced: corpus-transient-fatal-abort — client awaits each reply under
a deadline; transient-only palette (healing partitions, delivering
delays) so a RETRYING client always finishes; the buggy client
classifies a missed deadline as fatal and aborts the WHOLE session
(later timers step nothing, late replies dropped). Distinct from
demo-net-buggy: learns-and-misclassifies (blast radius) vs never-learns
(blindness).

Oracle session_complete: completed:<task> == true for all accepted;
detail names abandoned tasks.

Recall measured then pinned: found 79/100 at seed 0xD1CE, budget 100;
divergent 0; 21 fault-light universes PASS; sometimes session_aborted
reached.

Gate: "corpus recall gate: corpus-transient-fatal-abort" (exact exit 1

## Increment 4 (same PR): VB-010 resume-becomes-replay

Provenance: langchain-ai/langgraph issue #7361 (2026-03-31, 1.1.x
regression) — resume from a checkpoint_id re-executes from the
BEGINNING, not the interrupt point; the checkpoint was fine (dropping
checkpoint_id works around it). Verified at point of use.

Reduced: corpus-resume-replay — side-effecting steps + durably fsynced
progress cursor; on crash-recovery the cursor is READ BACK then ignored
(pipeline rescheduled from step 0). Crashes-only palette 0..=2 — the
first pin (count 1..=2) hit 100/100, which trips the anti-gaming rule
(a palette must not guarantee failure); widened to include crash-free
universes, which PASS. Duplication engine = crash-recovery itself
(distinct from VB-002 lossy-retry and VB-007 stale-sweep).

Oracle resume_at_most_once: applied:<step> <= 1 across crash epochs.
